# memU-server 测试指南

## 前置条件

1. 已安装 Python 3.13+
2. 已安装 uv 包管理器
3. 已配置 `.env` 文件
4. PostgreSQL 数据库正在运行

## 步骤 1：检查配置

确保 `.env` 文件配置正确：

```cmd
cd memU-server
type .env
```

应该看到：

```
# 通义千问 API 配置
OPENAI_API_KEY=sk-xxx
OPENAI_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
DEFAULT_LLM_MODEL=qwen-plus

# 数据库配置
DATABASE_URL=postgresql+psycopg://postgres:postgres@localhost:5432/memu
```

## 步骤 2：启动 PostgreSQL 数据库

如果使用 Docker：

```cmd
# 进入 memU-server 目录
cd memU-server

# 启动数据库
docker compose up -d postgres

# 检查数据库状态
docker compose ps
```

应该看到 postgres 容器正在运行。

## 步骤 3：启动 memU-server

在第一个终端窗口：

```cmd
# 进入 memU-server 目录
cd memU-server

# 启动服务（开发模式）
uv run fastapi dev app/main.py
```

## 步骤 4：运行测试

### 完整 API 测试（推荐）

**test_api.py** - 完整 API 测试套件
```cmd
# 在第二个终端窗口运行
cd memU-server
python tests/test_api.py
```

这个测试脚本会按顺序测试所有接口：
1. 健康检查 - 验证服务是否正常运行
2. POST /memorize - 存储对话记忆和系统观察
3. POST /retrieve - 检索相关记忆
4. POST /chat - 记忆增强对话
5. POST /proactive/analyze - 主动推理分析
6. POST /proactive/generate - 生成主动推理消息
7. POST /proactive/quick - 快捷主动推理
8. GET /proactive/cooldown - 获取冷却状态

## 常见问题

### Q1: 启动时报错 "OPENAI_API_KEY 未设置"

检查 `.env` 文件是否存在，并且包含正确的 API Key。

### Q2: 启动时报错 "数据库配置不完整"

检查 `.env` 文件中的数据库配置，确保 PostgreSQL 正在运行。

### Q3: 测试脚本报错 "无法连接到 http://127.0.0.1:8000"

确保 memU-server 正在运行。检查第一个终端窗口是否有错误信息。

### Q4: 测试返回 500 错误

查看 memU-server 终端的错误日志，通常是：
- API Key 无效
- 数据库连接失败
- LLM 调用失败

### Q5: 记忆存储成功但检索不到（重要）

这是最常见的问题，原因和解决方案：

**原因 1：向量化需要时间**
- memU 需要时间处理记忆并生成向量嵌入（embedding）
- 解决方案：等待 30 秒以上再进行检索测试

**原因 2：数据库中没有数据**
- 记忆可能没有真正存储到数据库
- 解决方案：运行数据库诊断脚本
  ```powershell
  uv run python tests/test_database_check.py
  ```

**原因 3：查询相似度不够**
- 查询文本和记忆内容的相似度太低
- 解决方案：使用更相关的查询文本，或检查嵌入模型配置

**原因 4：memU 判断不需要检索**
- 响应中 `needs_retrieval: False` 表示 memU 认为不需要检索
- 这可能是因为记忆还在处理中，或者查询不够明确

**调试步骤：**
1. 运行数据库诊断：`uv run python tests/test_database_check.py`
2. 检查是否有记忆项存储
3. 增加等待时间到 30 秒
4. 使用更具体的查询文本

### Q6: 测试超时 "Request timed out"

系统观察记录的处理需要调用 LLM 进行分析，可能需要较长时间（30-60秒）。测试脚本已设置 120 秒超时。如果仍然超时，检查：
- 网络连接是否正常
- 通义千问 API 是否可用
- API Key 是否有效

### Q7: 测试超时（远程数据库）

如果使用远程 PostgreSQL 数据库，向量搜索可能需要更长时间：

**症状**：
- `/retrieve` 接口超时（30-90秒）
- `/chat` 接口超时
- `/memorize` 首次调用超时

**原因**：
1. 远程数据库网络延迟
2. 向量化模型首次加载需要时间
3. pgvector 向量搜索在远程数据库上较慢

**解决方案**：
1. 使用本地 PostgreSQL 数据库（推荐）
2. 增加测试超时时间（已在 test_api.py 中设置为 90 秒）
3. 优化数据库连接池配置（已配置）

### Q8: Chat 接口返回 500 错误

检查 memU-server 终端的错误日志，常见原因：

**错误 1：LLM API 调用失败**
- 原因：API Key 无效或网络问题
- 解决：检查 `.env` 文件中的 API Key 和 BASE_URL

**错误 2：数据库连接失败**
- 原因：数据库连接超时或断开
- 解决：检查数据库是否正常运行，查看连接池配置

**错误 3：向量化失败**
- 原因：嵌入模型加载失败
- 解决：检查网络连接，确保可以访问模型 API

### Q9: Chat 接口不返回记忆

如果 `memories_used` 始终为 0：

1. **记忆还未向量化**：等待 30 秒以上
2. **查询相似度不够**：使用更具体的问题
3. **数据库中没有记忆**：先调用 `/memorize` 存储记忆
4. **检索被禁用**：确保 `retrieve_memories: true`

测试记忆检索：
```cmd
# 1. 存储记忆
curl -X POST http://127.0.0.1:8000/memorize -H "Content-Type: application/json" -d "{\"messages\":[{\"role\":\"user\",\"content\":\"我喜欢 Python\"},{\"role\":\"assistant\",\"content\":\"Python 很棒！\"}]}"

# 2. 等待 30 秒

# 3. 测试检索
curl -X POST http://127.0.0.1:8000/chat -H "Content-Type: application/json" -d "{\"message\":\"我喜欢什么编程语言？\",\"retrieve_memories\":true}"
```

### Q10: 主动推理不生成建议

如果 `/proactive/analyze` 返回 0 条建议：

**原因：冷却机制**
- 主动推理有冷却时间，防止频繁触发
- 疲劳提醒：30 分钟冷却
- 深夜提醒：1 小时冷却

**解决方案**：
- 测试时使用 `skip_cooldown: true` 参数
- 生产环境不要跳过冷却检查

```cmd
# 测试时跳过冷却
curl -X POST http://127.0.0.1:8000/proactive/analyze ^
  -H "Content-Type: application/json" ^
  -d "{\"context\":{\"fatigue_level\":\"Tired\"},\"skip_cooldown\":true}"
```

## 性能说明

### 首次调用较慢

首次调用以下接口会较慢（30-60秒）：
- `/memorize` - 需要初始化向量化模型
- `/retrieve` - 需要初始化向量化模型
- `/chat` - 需要初始化向量化模型和 LLM

后续调用会快很多（3-10秒）。

### 远程数据库性能

如果使用远程 PostgreSQL：
- 向量搜索会比本地慢 5-10 倍
- 建议使用本地数据库或优化网络连接

### 超时设置

test_api.py 中的超时设置：
- 健康检查：30 秒
- 存储记忆：60 秒（首次）
- 检索记忆：90 秒
- 对话接口：90 秒
- 主动推理：90 秒
