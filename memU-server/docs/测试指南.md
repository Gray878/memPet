# memU-server 测试指南

## 前置条件

1. 已安装 Python 3.13+
2. 已安装 uv 包管理器
3. 已配置 `.env` 文件
4. PostgreSQL 数据库正在运行

## 步骤 1：检查配置

确保 `.env` 文件配置正确：

```powershell
cd memU-server
type .env
```

应该看到：

```
# 通义千问 API 配置
OPENAI_API_KEY=sk-xxx
OPENAI_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
DEFAULT_LLM_MODEL=qwen-plus

# 数据库配置
DATABASE_URL=postgresql+psycopg://postgres:postgres@localhost:5432/memu
```

## 步骤 2：启动 PostgreSQL 数据库

如果使用 Docker：

```powershell
# 进入 memU-server 目录
cd memU-server

# 启动数据库
docker compose up -d postgres

# 检查数据库状态
docker compose ps
```

应该看到 postgres 容器正在运行。

## 步骤 3：启动 memU-server

在第一个终端窗口：

```powershell
# 进入 memU-server 目录
cd memU-server

# 启动服务（开发模式）
uv run fastapi dev app/main.py
```

成功启动后会看到：


## 常见问题

### Q1: 启动时报错 "OPENAI_API_KEY 未设置"

检查 `.env` 文件是否存在，并且包含正确的 API Key。

### Q2: 启动时报错 "数据库配置不完整"

检查 `.env` 文件中的数据库配置，确保 PostgreSQL 正在运行。

### Q3: 测试脚本报错 "无法连接到 http://127.0.0.1:8000"

确保 memU-server 正在运行。检查第一个终端窗口是否有错误信息。

### Q4: 测试返回 500 错误

查看 memU-server 终端的错误日志，通常是：
- API Key 无效
- 数据库连接失败
- LLM 调用失败

### Q5: 记忆存储成功但检索不到（重要）

这是最常见的问题，原因和解决方案：

**原因 1：向量化需要时间**
- memU 需要时间处理记忆并生成向量嵌入（embedding）
- 解决方案：等待 30 秒以上再进行检索测试

**原因 2：数据库中没有数据**
- 记忆可能没有真正存储到数据库
- 解决方案：运行数据库诊断脚本
  ```powershell
  uv run python tests/test_database_check.py
  ```

**原因 3：查询相似度不够**
- 查询文本和记忆内容的相似度太低
- 解决方案：使用更相关的查询文本，或检查嵌入模型配置

**原因 4：memU 判断不需要检索**
- 响应中 `needs_retrieval: False` 表示 memU 认为不需要检索
- 这可能是因为记忆还在处理中，或者查询不够明确

**调试步骤：**
1. 运行数据库诊断：`uv run python tests/test_database_check.py`
2. 检查是否有记忆项存储
3. 增加等待时间到 30 秒
4. 使用更具体的查询文本

### Q6: 测试超时 "Request timed out"

系统观察记录的处理需要调用 LLM 进行分析，可能需要较长时间（30-60秒）。测试脚本已设置 120 秒超时。如果仍然超时，检查：
- 网络连接是否正常
- 通义千问 API 是否可用
- API Key 是否有效

### Q7: Chat 接口返回 500 错误

检查 memU-server 终端的错误日志，常见原因：

**错误 1：`cannot import name 'get_llm_client'`**
- 原因：代码版本问题
- 解决：确保使用最新版本的 `main.py`，应该使用 `service._llm_client` 而不是导入 `get_llm_client`

**错误 2：`RetrieveMixin.retrieve() got an unexpected keyword argument 'query'`**
- 原因：参数名称错误
- 解决：应该使用 `text` 参数而不是 `query`

**错误 3：LLM 响应格式错误**
- 原因：不同 LLM 返回格式不同
- 解决：代码已处理 dict 和 string 两种格式

### Q8: Chat 流式响应中断或乱码

**问题 1：流式响应突然中断**
- 检查网络连接是否稳定
- 检查 LLM API 是否正常
- 增加超时时间

**问题 2：中文乱码**
- 确保使用 UTF-8 编码
- curl 命令添加 `-H "Accept-Charset: utf-8"`
- 检查终端编码设置

### Q9: Chat 接口不返回记忆

如果 `memories_used` 始终为 0：

1. **记忆还未向量化**：等待 30 秒以上
2. **查询相似度不够**：使用更具体的问题
3. **数据库中没有记忆**：先调用 `/memorize` 存储记忆
4. **检索被禁用**：确保 `retrieve_memories: true`

测试记忆检索：
```powershell

curl -X POST http://127.0.0.1:8000/memorize -H "Content-Type: application/json" -d '{"type":"conversation","content":[{"role":"user","content":{"text":"我喜欢 Python"},"created_at":"2024-02-15T10:00:00"}]}'

# 2. 等待 30 秒

# 3. 测试检索
curl -X POST http://127.0.0.1:8000/chat -H "Content-Type: application/json" -d '{"message":"我喜欢什么编程语言？","retrieve_memories":true}'
```
